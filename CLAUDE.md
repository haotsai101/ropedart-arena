## Agent roster

Three agents are available. Pick the right tier — over-spending on the senior agent for a string fix wastes tokens.

| Agent | Model | Use for |
|---|---|---|
| `godot-engineer` | Sonnet | New systems, multi-file features, bug diagnosis, anything needing `run_project` verification |
| `godot-junior` | Haiku | Mechanical single-file edits: string fixes, constant tweaks, adding a key handler, renaming a property. No run-verify. |
| `game-ui-designer` | Sonnet | HUD layout, menus, Tween animations, particle VFX, screen shake, visual polish |

**Decision rule:**
- Can you specify the exact file + line + change? → `godot-junior`
- Does it touch UI feel, animations, or visual feedback? → `game-ui-designer`
- Everything else → `godot-engineer`

**Fork instead of spawning** when the relevant files are already loaded in the conversation context and the task is diagnosis-only (no code changes needed). A fork shares the prompt cache and avoids re-reading files already in context.

## Running the game

Open Godot 4.7 (standard build, not .NET) → Import → select `project.godot` → F5.

There is no CLI build or test runner. All iteration happens in the Godot editor. The Godot MCP server (`godot` tool) is configured and can be used to inspect the scene tree and node properties without switching to the editor.

## Architecture: 2D logic, 3D rendering

**The invariant that governs every script**: all gameplay math (movement, collision, kill detection, rope dart flight) runs as `Vector2` on the XZ plane. `x` maps to world X, `y` maps to world Z. The 3D transform is purely for rendering. Never introduce vertical (Y-axis) gameplay.

`get_pos_2d() -> Vector2` on `player.gd` is the canonical way to read any player's gameplay position. All kill checks and bot targeting use this.

## Game systems and their wiring

**`GameManager` (autoload)** — `scripts/game_manager.gd`
- Singleton accessible as `GameManager` from any script.
- Owns the round state machine: `LOBBY → COUNTDOWN → PLAYING → ROUND_END → MATCH_END`.
- Spawns all player instances at startup via a 0-second `SceneTreeTimer` (fires next frame, after `change_scene_to_file` has fully applied). Players are added as children of `get_tree().current_scene`.
- Key exports: `total_players`, `human_count`, `bot_difficulty`, `lives_per_round`, `rounds_to_win`.

**`player.gd`** — `CharacterBody3D`, one instance per player
- Input: `player_index == 0` → keyboard (WASD/arrows/Space); `player_index >= 1` → gamepad `player_index - 1`. When `is_bot == true`, input is delegated to `bot_controller` via `get_desired_move()` / `get_desired_aim()` / `get_desired_throw()`.
- Movement is blocked (velocity zeroed) when `GameManager.current_state != PLAYING`.
- `_clamp_to_rope_leash()`, called right after `move_and_slide()` every physics tick: once the owner's rope dart is `ANCHORED`, clamps the player onto the circle of radius `DART_ROPE_LENGTH` around the dart's anchor point rather than letting them walk further away — a real tether limit, not just a visual one. `DART_STATE_ANCHORED` / `DART_ROPE_LENGTH` mirror rope_dart.gd's `State.ANCHORED` ordinal and `ROPE_LENGTH` by hand (no shared constant between the two scripts, same as `HITBOX_DEBUG_RADIUS`).
- Owns the rope itself, in two entirely different forms depending on `dart == null`, both driven from `_update_persistent_rope()` (called from `_process()`/`_physics_process()`):
  - **Idle** (`dart == null`): the old cheap look — `_rope_segments` (`ROPE_SEGMENTS` = 16 `MeshInstance3D` cylinders, built once in `_setup_dagger_in_hand()`) redrawn every frame as a tight coil around the forearm (`_render_rope_coiled()`, anchored at a `lowerarm.r` `BoneAttachment3D`). Purely kinematic/procedural — there's no obstacle to avoid while the rope is just sitting on the character's own arm.
  - **Thrown** (`dart != null`): a **real physics-simulated rope**, not scripted geometry. `_spawn_physics_rope()` builds a chain of `ROPE_PHYSICS_SEGMENTS` (8) `RigidBody3D` collision links (capsule `CollisionShape3D` only — no mesh of their own, see the VISUAL bullet below) between two kinematic (`freeze_mode = FREEZE_MODE_KINEMATIC`) endpoint bodies: a hand anchor and a tip anchor. `_update_physics_rope_anchors()`, called every `_physics_process()` tick, drives the hand anchor to `_get_rope_hand_anchor_pos()` and the tip anchor to the dart's actual rendered pommel position (`_get_rope_tip_target()`) — so Godot's own solver, not any script, is what keeps the rope off pillars/trees/cacti and makes it drape/rest against them when the owner ends up on the far side of an obstacle from an anchored dart. `_free_physics_rope()` tears the whole chain down once `dart` goes back to `null` (recall complete, kill, round reset).
  - **Joints are raw `PhysicsServer3D` pin joints (`_join_rope_pin()`, via `PhysicsServer3D.joint_make_pin()`), NOT `PinJoint3D` nodes.** This is a deliberate, root-cause-driven fix, not a stylistic choice: a first version used `PinJoint3D` nodes, whose single-shared-`global_position` setup implicitly derives ONE local anchor offset per body from wherever the joint node and both bodies happened to be at the moment the joint was configured. That version visibly rendered as short, disconnected dashes in a real user screenshot ("each segment completely unattached") — root cause, confirmed by explicitly measuring the real world-space gap at every joint (not just checking for engine errors, which showed none): every body was spawned clustered within ~0.16 units of the hand at identity rotation, so the auto-captured offsets landed near each body's own *center* rather than its true capsule *end*, meaning the solver was actually enforcing "keep body centers near each other," not "keep consecutive ends touching" — a permanent, steady-state wrong equilibrium, not just a fast-motion artifact (confirmed since the user's screenshot showed a stationary/`ANCHORED` dart). The fix: `PhysicsServer3D.joint_make_pin(joint, body_a, local_a, body_b, local_b)` takes each body's local anchor point as an **explicit, independent** argument, so every segment's true end (`Vector3(0, ±HALF_LEN, 0)`) and both anchors' own origin (`Vector3.ZERO`) can be declared correctly regardless of whether the bodies happen to coincide in world space when the joint is created — which they generally don't at throw-instant (dart starts at the hand, chain is laid out at its full fixed length), and that's fine: the mismatch is exactly the intended "rope pays out from the hand as the dart flies away" slack, not a bug. Segments are also now spawned with correct initial rotation (local Y aligned along the hand→tip layout direction) instead of defaulting to identity/pointing straight up.
  - **Joint bias/damping are left at `PhysicsServer3D`'s own defaults (0.3 / 1.0) — do not stiffen these without re-measuring.** A first attempt at addressing the disclosed fast-motion risk below tried tuning these stiffer (bias 0.9, damping 2.0); measured via a temporary per-tick gap probe, that config caused the chain to explode — the real per-joint gap grew *exponentially*, from single digits to tens of trillions of units within a couple seconds, on an otherwise-idle chain with no unusual input. Reverting to the defaults and re-measuring the same way showed gaps consistently settling to ~0.02–0.1 units at rest, with brief (few-tick) spikes to ~10–12 units at the instant of a throw that reliably decayed rather than diverging.
  - **Segments have `gravity_scale = 0.0` and are hard-locked to the dart's `plane_y` height every physics step (`scripts/rope_segment_body.gd`, attached to every dynamic segment via `_integrate_forces()`, NOT `PinJoint3D`/`gravity_scale` alone).** Root-caused from a real user SCREEN RECORDING (not a screenshot): once anchored, the rope visibly collapsed to a tiny ~0.3–0.5 unit stub next to the dagger within moments and stayed that way for the rest of the clip, sagging downward, even with the character standing several grid cells away. Root cause: default gravity (`gravity_scale = 1.0`, never overridden) pulled every light (`ROPE_SEGMENT_MASS = 0.03`) segment down every step, and the deliberately un-stiffened joint bias/damping (see above — stiffening was already tried once and found to explode the chain) only *softly* corrects position error each step, so it couldn't keep the chain taut against a continuous downward force across a real multi-unit span — it sagged into a different equilibrium instead. This is also why the PREVIOUS fix's own verification (a gap probe measuring only LOCAL adjacent-segment gaps, run for a few seconds right after a throw) never caught it: local gaps stayed small even while the whole chain sagged as a unit; nothing measured whether the chain's TOTAL reach still matched the real hand-to-dart distance over a sustained hold. Per the user's own explicit, verbatim requirement — "I want the rope to disregard gravity and live on a plane" — every segment now has `gravity_scale = 0.0` (the literal "disregard gravity") AND `rope_segment_body.gd`'s `_integrate_forces(state)` override forces `state.transform.origin.y` back to a fixed `locked_y` (read once from `dart.plane_y` at spawn — see `_get_rope_plane_y()`) and zeroes `state.linear_velocity.y`, every physics step — Godot's own documented, solver-safe way to override a `RigidBody3D`'s transform each step (poking `global_position` from outside physics processing, e.g. once per `_physics_process()` tick the way the kinematic anchors are driven, is explicitly discouraged by Godot's docs as producing "unpredictable behavior"). The kinematic hand anchor is ALSO now Y-locked to the same `plane_y` (`_get_rope_hand_anchor_pos()`, X/Z from the real animated hand bone but Y forced flat) rather than following the hand bone's real bob/animation height — otherwise the joint to the first (already Y-locked) segment would fight a small constant Y mismatch every single step. **Verified by direct measurement, not just no-errors**: a temporary probe printed the real hand-to-tip distance vs. the chain's actual reach (max distance from the hand anchor to any segment/tip) every ~1 second, specifically while `ANCHORED` (not just right after a throw), sustained across ~90 seconds total of bot play — chain reach tracked the real span continuously the whole time (e.g. one bot's anchored dart: real span shrank from 5.966→2.558 as it walked closer, chain reach followed in lockstep every sample), no collapse-toward-a-point ever reproduced. A separate simultaneous probe confirmed `max_y_drift` was exactly `0.0000` on every single sample. The only residual (expected, benign) deviation seen: on a short throw (small real span), chain reach can exceed the real span by 1–3 units — the fixed 8-unit rope has to go somewhere with gravity off, so it bulges/curves within the XZ plane rather than hanging straight — this is the disclosed "visible slack on close throws" behavior, not a bug, and is the *opposite* failure mode from the collapse this fix addresses.
  - **VISUAL: the thrown rope is rendered as one continuous tube mesh, NOT one `CylinderMesh`/capsule per physics segment.** Per an explicit follow-up user request ("Can the rope be rope instead of segments of bars?"), `_update_rope_tube_mesh()` (called every `_process()` frame from `_update_persistent_rope()`) rebuilds a single `MeshInstance3D` (`_physics_rope_tube_mesh`) each frame: it takes `[hand anchor, every dynamic segment's center, tip anchor]` as control points (same order as the joint chain), samples a Catmull-Rom curve through them at `ROPE_TUBE_CURVE_SAMPLES` (48) steps via `Vector3.cubic_interpolate()`, then extrudes a round `ROPE_TUBE_RADIAL_SEGMENTS`-sided (8) tube of constant `ROPE_RADIUS` along that curve via `SurfaceTool` (`_build_tube_mesh()`). This is a **pure rendering decoupling** — the underlying `RigidBody3D` segments, their capsule collision shapes, and the whole plane-lock/no-gravity mechanism above are completely unchanged; only what gets drawn from their positions changed, from N disjoint capsule meshes with visible seams at every joint to one smooth surface. `set_surface_override_material()` must be applied AFTER `mi.mesh` is assigned inside `_build_tube_mesh()` (not on the freshly-created, still-meshless `MeshInstance3D`) — Godot errors ("Index p_surface = 0 is out of bounds") on a mesh with zero surfaces.
  - **`_physics_rope_tube_mesh` MUST have `top_level = true`.** Root-caused from a real user screen recording, frame-analyzed directly (not just a screenshot): while the character stood still, a dark rope-like curve floated in open space near them, disconnected from both the character and the dart, reshaping into a visibly different distorted curve EVERY SINGLE FRAME (an open curl, then a tight C-arc, then a zigzag, then an S-curve). A separate live playtest report the same round — "the dart is flying correctly but the rope is not" — turned out to be the exact same bug, just observed during FLYING instead of after a throw ended. Root cause: the tube mesh's control points (`_update_rope_tube_mesh()`) are all read from `.global_position` — already WORLD-space coordinates — but the `MeshInstance3D` itself was a plain (non-`top_level`) child of `self` (the player, a moving/constantly-rotating `CharacterBody3D`), so Godot rendered its already-global vertex data through `player.global_transform * vertex` — double-transforming it. The result visibly rotates/warps in lockstep with the PLAYER's own facing direction (which changes continuously while aiming/tracking a target, via `_facing_dir`/`aim_dir`), not with the rope's real physics position — exactly matching a reshaping curve unconnected to the actual chain. Fix: `mi.top_level = true`, set once right after the `MeshInstance3D` is created in `_update_rope_tube_mesh()` — this makes the node's `global_transform` NOT inherit from its parent at all, so the already-global vertices render at their literal positions with no further transform needed. (The underlying physics bodies were never affected by this bug — they're parented under `_physics_rope_root`, a separate non-moving `Node3D`, specifically to avoid this exact class of problem; only the tube-mesh rendering code added in the previous round missed applying the same principle.) The mesh is still a persistent child of `self` created once (not of `_physics_rope_root`, which gets torn down and rebuilt every throw) so its own node lifecycle stays independent of the physics chain's — `top_level` only changes how its transform is computed, not its parent/ownership.
  - **A second hypothesis raised the same round — an orphaned/leftover physics chain from an incomplete `_free_physics_rope()` — was investigated and NOT found to be a real bug.** Every path that sets `dart = null` (`kill()`, `reset_for_round()`, `_on_dart_returned()`) is reachable from `_process()`'s unconditional `_update_persistent_rope()` call (which runs before any `is_dead`/`is_falling` early-return), and `_update_persistent_rope()` itself calls `_free_physics_rope()` whenever `dart == null and _physics_rope_active` — so cleanup fires on every code path, not just the “normal” recall/pickup one. Verified via a temporary scene-tree census (counting nodes named `PhysicsRopeChain_<player_index>` under the arena root) sustained across ~2400 frames (~40s) of hard-bot rapid-fire throw/recall/pickup cycling across 3 simultaneous players — chain count never exceeded 1 per player. The visual symptom the user saw was fully explained by the `top_level` bug above; this hypothesis, while reasonable given the visual evidence alone, was not the actual cause.
  - **Verification note**: confirmed by actually measuring real world-space quantities every tick via temporary probes (never just "no errors printed," which had already failed to catch bugs from earlier rounds) — joint gap, then chain-reach/Y-drift, then (this round) a scene-tree orphan census plus a `mesh_basis_identity`/rotation-decoupling check sampled every few frames across many dozens of real throw cycles. **Still not verified visually** — no screenshot/video access in this session, so whether the tube mesh now actually renders as a stable, correctly-positioned rope on screen (rather than just being numerically decoupled from the player's rotation, which is what was directly measured) remains unconfirmed pending a fresh screenshot or recording.
  - **Known, disclosed limitation**: the chain is simulated for the dart's *entire* time out (FLYING/ANCHORED/RECALLING alike), including while the tip anchor is being dragged at up to 18–36 units/sec during a fast throw or recall. Measured gap spikes during this (~10–12 units, decaying within a few ticks, from an earlier fix round's own probe) never diverged, but how that transient actually looks on screen — now through the tube mesh — is unverified.   - **UNSPOOL/"reeling out" FIX (per explicit user direction, from a real screen recording reviewed frame-by-frame — not a screenshot): the rope no longer appears pre-extended to its full length the instant a throw fires.** Root cause, confirmed by the coordinator's own frame review before any code changed: `_spawn_physics_rope()` used to lay every one of the 8 dynamic segments out along `span_dir` to the chain's full fixed `ROPE_PHYSICS_SEGMENT_LENGTH`-per-segment layout *regardless of the dart's real (usually near-zero, right at throw-instant) distance from the hand* — correctness never depended on this (the zero-baked-in-offset joint design means each joint's local anchors are declared independently of the bodies' actual world positions at spawn), but the visual result was exactly the user's complaint: the whole 8-unit rope appeared at once, already long, with only the *unused* length showing as slack/droop, never visibly growing out of the hand as the dart travels further. Fix: segments are now spawned **bunched near the hand** instead (`ROPE_BUNCH_SPACING` = 0.06, only a few centimeters apart, in `player.gd`'s `_spawn_physics_rope()`), letting the kinematic tip anchor (already tracking the dart's live position every tick, unchanged) drag the chain out progressively as joint constraint violations resolve tick by tick, rather than laying the whole capacity out up front.
    - **This intentionally reintroduced the same class of instability risk the codebase had already flagged and rejected a stiffer fix for once before (see `ROPE_PHYSICS_*`'s "bias 0.9 explodes the chain" note above) — and it was real, not hypothetical.** Verified via a temporary per-tick probe (printing real hand-to-dart distance vs. the chain's own actual max reach from the hand, sampled for the first 40 physics ticks after every throw, across dozens of real bot throws): with bunched spawn and NO additional mitigation, the chain reliably underwent a genuine multi-tick "crack the whip" resonance — `chain_max_reach` shot up to **~4x the total rope length** (30+ units against a real hand-to-dart distance of ~4, one single tick alone moving a segment ~7 units / ~420 units/sec) before slowly decaying back down over another 20–30 ticks. Root cause: with every one of the chain's 9 joints simultaneously violated by ~1 unit at spawn (each capsule's own local anchor point sits `ROPE_PHYSICS_SEGMENT_HALF_LENGTH` from its near-coincident neighbor's, even though the body *centers* are bunched together), Godot's iterative Gauss-Seidel joint solver propagates the correction across the whole 9-joint chain within a single physics tick, and on very light (`ROPE_SEGMENT_MASS` = 0.03) bodies that compounds link-to-link instead of damping.
    - **Fix that worked, verified by re-running the same probe**: a hard XZ speed clamp on every dynamic segment, `MAX_SEGMENT_SPEED` = 45.0 (`rope_segment_body.gd`, enforced every physics step inside the existing `_integrate_forces()` plane-lock override — the same Godot-documented, solver-safe entry point already used for the Y-lock, not a new mechanism). This brought `chain_max_reach` down to a bounded ~7–9 units max (never far past the chain's own 8-unit capacity) that converges to closely track the real hand-to-dart distance within roughly 10–20 ticks, reproduced stably across many dozens of throws with 3 simultaneous bots continuously throwing. **Two alternative levers were tried first and explicitly rejected, kept as notes in the code so they aren't retried blindly**: (1) `PhysicsServer3D.PIN_JOINT_IMPULSE_CLAMP` was considered but not needed once the velocity clamp alone worked; (2) explicitly setting pin-joint bias *lower* than the default 0.3 (down to 0.04, on the theory that bias directly scales per-step positional correction) was tried and measured, via the same probe, to make the overshoot *worse* (spiked past 12 units) — a weaker bias makes each joint hold its own pair of bodies together less firmly, so the chain resists cascading disturbance from its other 8 joints *less*, letting the effective total length balloon further, not less. Bias/damping are left at `PhysicsServer3D`'s defaults, unchanged from the prior round.

  - **ROUND 2 UNSPOOL FIX — "growing leash" position clamp (per direct written user feedback: "The rope is not uncoiling the way it should" — the round-1 fix above was insufficient on its own).** Both previously-flagged-but-untried levers were actually tried this round, in isolation, with the same per-tick probe, and neither meaningfully fixed the remaining problem: (1) a much smaller `ROPE_BUNCH_SPACING` wasn't the bottleneck (the initial layout's own spacing barely matters — the dominant contributor is each capsule's own half-length anchor offset, ~0.5 units, which doesn't shrink with tighter bunching); (2) `MAX_SEGMENT_SPEED` tested at 8.0 (far below the shipped 45.0), in isolation at default bias, barely changed the chain's overall unspool timing at all — it still reached ~70–90% of full length within the same ~4–6 ticks. **Root cause of why a per-body speed clamp alone can never fully fix this**: it only bounds how fast any ONE body moves, but multiple bodies along the chain can each move at a modest, individually-legal speed *simultaneously*, and their combined effect still unfolds a large fraction of the chain's total slack within a handful of ticks — the real bottleneck is the chain's *aggregate* growth rate, which no per-body speed limit can bound on its own. Also investigated and ruled out as the dominant cause (per the coordinator's explicit request to check the render path, not just assume it's fine): the tube mesh's Catmull-Rom interpolation (`_update_rope_tube_mesh()`/`_build_tube_mesh()`) *does* measurably overshoot past the raw physics control points' own extent — verified via a temporary probe comparing the raw control points' max reach from the hand against the sampled curve's max reach — but the overshoot was small and secondary (typically 0.01–0.3 units, under ~4% of total rope length), not the dominant driver; the physics chain's own single-tick jump (still ~7 units within a few ticks even with the round-1 clamp) was the real problem.
    - **The actual fix**: `rope_segment_body.gd`'s `_integrate_forces()` now enforces a live, per-tick **position** cap (`max_reach_from_hand`, `hand_pos_2d`) on top of the existing Y-lock and speed clamp — any segment the solver tries to push further than `max_reach_from_hand` from the hand gets pulled back radially (preserving its current angle, so it doesn't fight the solver's own XZ shaping/obstacle-draping), with the outward-radial component of its velocity zeroed too so it doesn't immediately re-violate the cap and jitter against it. Critically, `max_reach_from_hand` is **not a fixed constant or a separate timer/ramp** — `player.gd`'s `_update_physics_rope_anchors()` recomputes it every physics tick directly from the REAL, already-known hand-to-dart distance plus a small fixed slack allowance (`ROPE_UNSPOOL_SLACK` = 1.0), capped at the chain's own total capacity (`DART_ROPE_LENGTH`). This ties the rope's visible/simulated extent directly and deterministically to how far the dart has *actually* traveled (itself already smooth and non-explosive, being a kinematic read of the dart's own position) instead of hoping the joint solver's emergent equilibrium happens to grow at a similar rate.

A new probe (max perpendicular deviation of every physics control point from the straight hand-to-tip line) showed this did NOT reliably help — in several real throws the deviation got WORSE at 0.4 than at 1.0 (e.g. one throw: deviation grew from ~1.5 to ~2.7 for a similar real distance). Root cause: `max_reach_from_hand` bounds distance from the HAND POINT in any direction (a sphere), not distance from the hand-to-tip LINE — and the chain's total physical length is fixed at 8 units (`DART_ROPE_LENGTH`) regardless of the real span, so for most of a throw there's always several units of "must go somewhere" excess capsule length; shrinking the sphere's radius doesn't reduce that excess, it just forces the same excess to fold into a smaller sphere, which can bulge MORE per unit of available room, not less. `ROPE_UNSPOOL_SLACK` was reverted to its original 1.0 — its role is purely round 2's unspool pacing, not tautness.
    - **The actual fix**: a second, independent clamp — `rope_segment_body.gd`'s `max_perp_from_line`/`tip_pos_2d` (new fields, alongside the existing `max_reach_from_hand`/`hand_pos_2d`) — caps each segment's perpendicular distance from the straight hand-to-tip line itself (a "tube" around the taut line), not distance from a point. `player.gd`'s new `ROPE_TAUT_PERP_RADIUS` (0.3) constant is pushed live every physics tick from `_update_physics_rope_anchors()`, same pattern as the existing radius-from-hand clamp: any segment beyond that perpendicular tolerance is pulled back toward the line (preserving its along-line position), with the perpendicular component of its velocity zeroed to prevent jitter. This is a plain position clamp on an already-computed transform — the same class of mechanism as the existing Y-plane lock and radius-from-hand clamp, not a change to the solver's own joint stiffness, so it doesn't carry the joint-bias explosion risk.
    - **Verified via the same probe, re-run after this fix**: max perpendicular deviation, which previously ranged ~0.5–2.7 units across real throws (frequently exceeding the actual hand-to-dart distance itself), now converges to roughly 0.3–0.4 (matching `ROPE_TAUT_PERP_RADIUS`'s own value almost exactly) within a few ticks of each throw and stays there — a dramatic, directly-measured reduction, not an assumed one. One residual, disclosed transient: a brief 1-tick spike (observed once, to 1.8, in one throw at tick 4–5, matching the timing of the round-2 growing-leash's own already-accepted brief transient) that decays to the ~0.3–0.4 steady band within 1–2 ticks — consistent with the existing precedent of "brief solver-driven spikes that decay quickly are acceptable, sustained/growing ones are not." No errors or divergence observed across an extended soak (dozens of throws, ~25s of continuous bot play).
    - **Honest confidence caveat**: numerically verified only (perpendicular deviation from the true taut line, directly measured before/after) — no screenshot/video access this session. The mechanism is now a much more literal match for "tension" (directly bounding how far the rope can bow off the straight line) than round 2's radius-from-hand approach, which this round's own measurement showed does NOT reliably control straightness at all despite intuition suggesting it would — a concrete lesson for any future tautness-related feedback on this system: reach for a distance-from-the-LINE clamp, not distance-from-a-POINT, and re-measure rather than assume. `ROPE_TAUT_PERP_RADIUS` (0.3) is, like `ROPE_UNSPOOL_SLACK`, a pure aesthetic tuning constant with no physics-derivable "correct" value — if the next feedback is "still too loose" or "now too stiff/straight," this is the first constant to adjust, and per this round's own lesson, should be re-verified with the same perpendicular-deviation probe rather than assumed.
- Signals: `player_killed(player)`, `player_eliminated(player)`. Both HUD and GameManager connect to these.
- `reset_for_round(lives, pos)` is the only way to restore a player between rounds.

**`rope_dart.gd`** — `Node3D`, spawned by player on throw
- The rope+dart system lives in one fixed horizontal plane at the owner's hand height: `plane_y` is computed once in `launch()` from `owner_player.get_hand_world_position().y` and held constant for the dart's whole lifetime, regardless of state or the owner's later movement. This is what lets its obstacle detection use *real* Godot physics (see below) while staying compatible with the rest of the game's flat-XZ-plane gameplay math.
- State machine: `FLYING → ANCHORED → RECALLING`. The rope has a fixed length (`ROPE_LENGTH`, 4× a character's height, ~8.0 units) rather than one that scales with charge — charge now scales `travel_speed` only. Thrown, it flies in a straight line until it hits a player, a real obstacle, the arena boundary, **or reaches `ROPE_LENGTH` without hitting anything** — all four cases now anchor it in place (`_anchor()`), stuck until the owner recalls it or walks over it (`pickup_radius`). **Reversed in this round, per explicit user direction**: reaching max range used to auto-trigger `recall()` (a "yank back" snap-taut, never actually anchoring in open air) instead of anchoring — the FLYING branch's `elif head_2d.distance_to(origin_2d) >= ROPE_LENGTH:` case in `_physics_process()` now calls `_anchor()` there too, matching the other three stop conditions exactly. Checked for stale assumptions elsewhere before considering this done: `bot_controller.gd`'s dodge logic only keys off `dart.state == FLYING` (stops dodging once anchored regardless of *why*, so it needed no change), and `player.gd`'s `_clamp_to_rope_leash()` already keys off `ANCHORED` generically (so a max-range-in-open-air anchor now correctly becomes a real tether point too, not just obstacle/player anchors). Pressing throw again while it's out (`dart != null`) still calls `recall()` manually; walking up to it (`pickup_radius`) still retrieves it in any state.
- Player hit detection is pure 2D math — no physics collision shapes involved: `_check_hits()` checks distance from `head_2d` to each non-owner player's capsule (`_seg_dist(head_2d, base, _capsule_top(base)) < hit_radius`), then distinguishes a tight `head_hit_radius` at the top of the capsule (instant `kill()`) from anywhere else along it (non-lethal `trip()` "clothesline" stagger). Either kind anchors the dart at the point of contact.
- Obstacle stopping during FLYING uses a **real physics raycast** each frame (`_raycast_obstacle`, at the fixed `plane_y`, over the frame's swept motion) against the map's actual `CollisionShape3D` geometry — the same collision pillars/trees/cacti already use against players — via `PhysicsDirectSpaceState3D.intersect_ray`, with every player body excluded by RID so the rope dart never anchors on a character. Plus a hard clamp at `arena_half` for the arena boundary. `plane_y` (see above) is clamped to `[MIN_PLANE_Y, MAX_PLANE_Y]` = `[0.5, 1.6]`, comfortably inside the `[0.0, 2.0]` Y-range every obstacle's collision box shares — the raw hand-bone sample can otherwise land outside that band mid-animation-swing and make the raycast miss real obstacles entirely. On a real-obstacle hit, `head_2d` is offset by `ANCHOR_EMBED_DEPTH` (-0.1175, i.e. pulled back, not pushed in) from the raycast's surface point — the dagger's own origin sits much closer to its pommel (local Z=+0.315) than its tip (local Z=-0.55), so for the dagger's true midpoint to land on the surface (half embedded, half visible) the origin has to sit slightly outside it, not past it.
- `_anchor()` freezes `_anchor_dir_2d` (a copy of `dir_2d` at that instant) — `_render()` uses it for the blade's orientation for the rest of `ANCHORED`, instead of continuously re-deriving it from the owner's current position (which made an anchored/embedded dart visibly swivel as the owner walked around it).
- `launch()` calls `_render()` once immediately at the end, synchronously — without it, `head_mesh` sits at its scene-default transform (world origin, the map's center) for the one render frame between `add_child()` and this dart's first `_physics_process()` tick.
- The rope itself is no longer this script's — see `player.gd`'s `_update_persistent_rope()` and its physics-chain doc comment above. This script's own `_raycast_obstacle()` (below) is still what stops the *dart head's* flight; the rope trailing behind it is now a fully separate, really-simulated `RigidBody3D` chain that reacts to obstacles on its own via Godot's physics solver, not anything this script computes.
- A player can only have one rope dart out (in flight, anchored, or recalling) at a time — `player.gd` only lets `_throw()` fire while its own `dart == null`.
- Adds itself to the `"darts"` group in `launch()` so `bot_controller.gd` can find incoming darts to dodge.
- Animations (see `player.gd`'s `ANIM_SOURCES`/`combat_moves.glb`/`spell_cast.glb`, retargeted from Quaternius's Universal Animation Library via Blender): `Sword_Attack` on a lethal melee slash (rope dart in hand), `Punch_Jab` on a non-lethal melee kick (rope dart thrown) — both one-shot, tracked in `ONE_SHOT_ACTION_CLIPS`, unchanged by the throw/recall rework below. Throw and recall are each their own 3-phase **Enter → Hold → Exit** sequence (`ThrowAnimPhase` enum; `_throw_anim_phase`/`_recall_anim_phase` + matching timers), replacing the old single one-shot `Spell_Simple_Shoot` clip and the old bare-loop `Push` clip respectively:
  - **Throw**: `Spell_Simple_Enter` (Enter, plays once) → `Spell_Simple_Idle` (Hold, held for as long as the just-thrown dart is still actually `FLYING` — `DART_STATE_FLYING` mirrors `rope_dart.gd`'s `State.FLYING` ordinal by hand, same convention as `DART_STATE_ANCHORED`) → `Spell_Simple_Exit` (Exit, plays once, then the phase returns to `NONE`). Triggered in `_throw()`.
  - **Recall**: `Spell_Simple_Enter` (Enter) → `Push` (Hold, held for as long as `_is_recalling` stays true — unchanged clip/meaning from before this feature, still reads as "reeling in") → `Spell_Simple_Exit` (Exit). Triggered alongside `dart.recall()`/`_is_recalling = true` in `_physics_process()`'s throw-again-to-recall branch.
  - **Phase advancement is time-driven** (`_advance_throw_anim()`/`_advance_recall_anim()`, called unconditionally every `_process()` frame), not driven by `AnimationPlayer.is_playing()` — each phase's real length is read once via `_anim_clip_length()` (`Animation.length` on the merged library, post the glTF importer's `_Loop`-suffix-stripping/auto-loop convention, e.g. `Spell_Simple_Idle_Loop` source clip → `Spell_Simple_Idle` imported name, same as `Push_Loop` → `Push`). This decoupling exists specifically so movement can interrupt the DISPLAYED clip without derailing the sequence's own progress tracking (see next bullet) — a single shared `AnimationPlayer` can only ever show one clip at a time, so if display and progress were the same mechanism, cutting to `Walking_A` mid-sequence would corrupt the phase's own advancement.
  - **Movement always wins, per explicit user requirement**: in `_process()`'s locomotion selection chain, `_is_dashing`/`is_moving` (Running_A/Walking_A) now sit ABOVE both `_recall_anim_phase`/`_throw_anim_phase` — if the player starts moving mid-sequence, the displayed clip cuts to the movement clip immediately; the phase timers keep advancing unseen in the background and resume displaying correctly once movement stops. **Documented trade-off**: because progress is time-driven rather than clip-position-driven, a phase interrupted by movement always resumes by restarting its own current clip from frame 0 (not mid-playback) once movement stops — chosen deliberately for correctness/simplicity over frame-perfect resumption. This is a reversal from the old behavior, where `_is_recalling`'s bare `Push` sat ABOVE movement in the same chain (recall used to keep looping uninterrupted while walking around freely). `_is_charging`/melee's `action_playing` guard keep their old relative position above movement — out of scope for this change and neither can actually co-occur with movement in practice anyway (charging zeroes velocity; melee wasn't reported as an issue).
  - Both phase/timer pairs are reset to `NONE`/`0.0` in `reset_for_round()` (new-round reset) and in `kill()` (a kill mid-throw/mid-recall force-frees `dart` directly, bypassing `_on_dart_returned()` — the only other place `_is_recalling` is cleared — so without this the sequence could get stuck displaying `Push`/`Spell_Simple_Idle` forever post-respawn with no dart to justify it).
  - `spell_cast.glb` (`Spell_Simple_Enter`/`Spell_Simple_Idle_Loop`/`Spell_Simple_Exit`) was retargeted in its own separate Blender pass, run **headless** (`blender --background --python <script>`, not the interactive Blender MCP bridge, which crashed on this project's `UAL1_Standard.glb` import — a `bpy.context.object` `AttributeError` deep in Blender's own glTF importer, reproducible outside any `temp_override`) — same world-space Copy Rotation constraint + `bpy.ops.nla.bake` technique as `combat_moves.glb`, with a freshly-reconstructed 16-bone map (hips/spine/chest/head + both arms + both legs; no bone map from the original `combat_moves.glb` retarget was ever saved to disk) since only 3 additional clips were needed. Exported as its own small glb rather than appended into the already-working `combat_moves.glb`, so the clips already retargeted there are never touched by this work.

### Rope dart move design reference

Three named moves, from design notes — current code implements Throw (Linear Strike) and Retrieval (Reel In) as described; the Wrap (Grapple/Bind) is not yet implemented (no bind/wrap mechanic exists today — a body hit currently applies `trip()`'s stagger, not a spiral bind).

- **The Throw (Linear Strike)**: The character delivers a directional thrust. The metal dart fires straight from the hand. The rope uncoils rapidly into a taut line. The asset maintains absolute linear rigidity at peak extension. Upon impact, the dart freezes briefly in place. The rope sags slightly to indicate lost momentum.
- **The Wrap (Grapple/Bind)**: The character initiates a wide, circular pivot spin. The dart follows a curved, high-velocity arc. The weapon makes contact with the target geometry. The dart rotates tightly around the impact point. The rope wraps around the object in spiral layers. The line snaps completely taut to lock the bind.
- **The Retrieval (Reel In)**: The character snaps their anchor hand backward. The dart dislodges instantly from the target. The weapon flies backward along the original trajectory line. The rope retracts dynamically toward the character's hand. The loose slack forms smooth, looping folds during recovery. The dart snaps back into the idle hand position.

**`bot_controller.gd`** — `Node`, attached as child `"BotController"` under bot players by GameManager
- Sets `parent.bot_controller = self` in `_ready()`.
- State machine: `CHASE → AIM → RETREAT`. Medium/Hard bots also dodge incoming darts via `_get_dodge_dir()`.
- `get_desired_throw()` consumes a one-shot `_throw_pending` flag; the player's `_prev_throw` tracking handles just-pressed detection correctly.

**`hud.gd`** — `CanvasLayer`, instanced from `scenes/hud.tscn` in main scene
- All UI nodes are created in code (no scene editor layout). `_build_skeleton()` creates the panel structure; `_setup_player_panels()` (deferred one frame) populates them after GameManager has spawned players.
- Connects to `GameManager.state_changed`, `round_ended`, `match_ended`, and each player's `player_killed` / `player_eliminated`.

**`arena_camera.gd`** — script on `Camera3D` in main scene
- Orthographic projection. Computes AABB of all alive players (by `lives > 0`) each frame and lerps `size` and position to keep them in frame.

**`arena_obstacle.gd`** — `StaticBody3D` script on pillar nodes
- Adds node to `"obstacles"` group. Exposes `get_rect_2d() -> Rect2` for the rope dart's swept-rect stop test — a plain 2D check, no physics-engine query needed.
- Also tags itself with an additional collision layer bit, `layer 2` (named `"rope_obstacles"` in `project.godot`'s `[layer_names]`), on top of whatever layer it already had (normally the default layer 1, left untouched) — this is the dedicated bit `player.gd`'s physics rope chain uses as its `collision_mask`, so the chain reacts to real obstacle geometry without ever being able to detect/collide with players, the ground, or the dart head (none of which carry this bit). One-directional by design: the chain's own `collision_layer` is left at `0`, so nothing else's mask can ever detect the chain back — this is what keeps a literally-simulated rope from leaking into gameplay (pushing a player, blocking movement, or interfering with the dart's own flight raycast).

## Duck typing convention

Player script properties (`player_index`, `is_dead`, `lives`, `get_pos_2d()`, `kill()`, etc.) are accessed from other scripts via duck typing on untyped variables. Use `for p in array:` (not `for p: Node in array:`) when iterating players, and `Variant` for signal callback parameters that receive player nodes, to avoid GDScript strict-mode type errors.
